# HarnessX command reference

Every command is invoked as `harness <verb> [subverb] [flags]`. The
command tree is grouped below by the paper layer it serves
(see [`PAPER-MAPPING.md`](PAPER-MAPPING.md) for the §-by-§ mapping).
This document is generated by hand from the source and updated whenever
a new command lands.

Run `harness --help` for an auto-generated list.

---

## Bootstrap and project lifecycle

### `harness init [--git] [--all] [--git-branch <name>]`
Writes `.harness/` (config, db, hooks, gitignore) into the current
directory. With `--git`, runs `git init` when no repo exists. With
`--all`, also registers the project in the cross-project workspace
registry. Idempotent.

### `harness new [stack] [path] [--yes] [--with-deps]`
One-command bootstrap: `git init` + `harness init` + `harness scaffold
apply <stack>` + `harness install-git-hooks`. Without `--yes` it prompts
for stack (`go|python|rails|react|ruby|rust`) and target path. With
`--yes` it runs non-interactively.

### `harness scaffold list|show <lang>|apply <lang> [--apply] [--with-deps] [--with-git]`
Deterministic language scaffolds (no LLM call). `apply` defaults to
dry-run; pass `--apply` to write files. Writes `.harness/config/
project.yaml` recording the canonical `test`/`lint`/`run` commands so
the wrappers below can resolve them.

### `harness install-git-hooks [--hooks pre-push,pre-commit,commit-msg|all] [--force]`
Installs HarnessX-managed git hooks. Defaults to `pre-push` only.
Refuses to overwrite foreign hooks unless `--force` is supplied. Bypass
flags at commit/push time:
- `HARNESS_SKIP_CI=1` skips pre-push
- `HARNESS_SKIP_PRECOMMIT=1` skips pre-commit
- `HARNESS_SKIP_COMMITMSG=1` skips commit-msg

### `harness uninstall`
Removes harness + state with confirmation.

### `harness update`
Self-updates via the configured release channel.

### `harness version`
Prints version + commit + build date.

### `harness doctor`
Diagnoses host environment: required tools, LSP servers, supply-chain
helpers, registered agents, project root, database state. Cites the
install command that would fix each red row.

## Project commands (resolved per-stack via `project.yaml`)

These wrappers look up the resolved command in `.harness/config/
project.yaml`. When the file is missing, they fall back to the bundled
default for the detected stack (`go.mod`, `pyproject.toml`,
`Cargo.toml`, …).

### `harness test [-- extra args]`
Runs the project's test command.

### `harness lint [-- extra args]`
Runs the project's linter.

### `harness dev [-- extra args]`
Runs the project's dev server (e.g. `uvicorn`, `vite dev`, `cargo run`).

### `harness bench [-- extra args]`
Runs the project's benchmark suite.

### `harness profile mem|cpu [-- extra args]`
Runs the memory or CPU profiler for the stack.

### `harness coverage [--threshold 0.9] [--pkg ./...]`
Runs `go test -cover` and gates the average coverage against the
threshold (default 90%). Exits non-zero on regression. Implements
paper §3.4.4 verification through deterministic sensors for Go
projects.

## §2 Interface — reasoning / acting / environment

### `harness ask "<prompt>" [--budget-usd]`
One-shot question answered by the default planning adapter. No
execution.

### `harness plan "<prompt>" [--budget-usd]`
LLM-backed plan + spec generation. Writes to `.harness/runs/<id>/`.
For the **deterministic** plan-as-contract artifact see `plan write`.

### `harness plan write "<prompt>" [--file path]* [--invariant text]* [--validate cmd]* [--rollback cmd] [--risk low|medium|high]`
Writes `.harness/artifacts/plans/PLAN-<ulid>.md` with intent, file
scope, invariants, validation commands, rollback, and risk tier — no
LLM call. Implements paper §3.4.2 *planning as contract formation*.

### `harness plan check --plan <id>`
Compares the current working-tree diff against the plan's file scope
and emits violations. Drives the `plan_scope` sensor (§3.4.4).

### `harness explain "<prompt>"`
Walks the user through the harness reasoning for a prompt.

### `harness context [build|show|providers]`
Inspects the deterministic context pack assembled before each LLM
call: memory entries, git changes, ripgrep hits, test map, LSP
references. Implements paper §3.2.6 context compaction.

## §3 Mechanisms — planning, memory, tools, control, optimisation

### `harness do "<prompt>" [--autonomy ask|strict|safe_execute|...] [--agent claude|codex|...] [--apply]`
Decomposes a prompt into a task graph and routes each task to its best
adapter. Default autonomy is `safe_execute`. With `--apply`, the diff
is written; without it the run stays dry. Outputs land at
`.harness/runs/_do/do-<timestamp>.md`.

### `harness loop [--max-attempts N] [--budget-usd] [--lint] [--test] [--apply]`
Watches the working tree, re-runs `harness do` on change, gates with
sensors, exits on green or attempts exhausted. Resume via
`harness loop resume <run-id>`.

### `harness ship "<prompt>" [--max-attempts] [--rate-limit-retries N] [--rate-limit-wait DURATION] [--base develop] [--branch-prefix feature] [--plan <id>] [--autonomy ask] [--dry-run] [--skip-commit]`
Single-command SDLC driver:
1. Branches `feature/<slug>` from `--base`.
2. Writes the spec via `harness feature`.
3. Up to `--max-attempts` iterations of `harness do` + `harness ci`.
4. Detects HTTP 429 in stderr and backs off exponentially before
   delegating to the next adapter in the router chain.
5. If `--plan <id>` is supplied, runs `planscope.Check` before
   committing and refuses commits with out-of-scope file edits.
6. Conventional-commit on success.

Combines paper §3.1.4 (orchestration), §3.4 (PEV loop), §3.4.2 (plan
contract), §3.5 (rate-limit-aware adaptation).

### `harness chat [--goal dev|ads|research|ops] [--adapter <id>] [--model <id>] [--step-timeout DURATION]`
Interactive REPL. Each prompt is sent to the planner, which returns a
typed JSON plan; the plan is executed deterministically against the
project. With `--adapter`, the planner runs through the named agent
(e.g. `--adapter ollama` for a local LLM); without it, a deterministic
template is used. Slash-commands inside the session:
- `/goal <name>` — switch the session goal
- `/plan <prompt>` — emit JSON plan without executing
- `/help` — list commands
- `/exit | /quit` — leave

### `harness feature "<prompt>" [--yes] [--budget-usd]`
Writes a feature spec into `.harness/artifacts/specs/`. Lightweight
LLM call.

### `harness bugfix`, `harness report`, `harness design-to-product`
Stage-specific workflows wrapping the same router. See `--help`.

### `harness flow list|show|apply|init <name>`
Bundled deterministic flows (e.g. `python-fastapi`). Each flow is a
phase sequence: deterministic steps + optional LLM phases. Gates
block downstream phases on red.

### `harness orchestrate list|show|run <name> [--dry-run] [--adapter-timeout DURATION]`
Reads an orchestration spec from `.harness/orchestrations/<name>.yaml`
and runs it. Steps declare a `role` (Manager/Planner/Coder/
Reviewer/Tester), a `command`, and/or an `adapter`. Adapter steps
dispatch to the named agent with the previous blackboard entries as
context. A blackboard JSON is written to `.harness/artifacts/
runs/<id>/blackboard.json`. Implements paper §4.1.1 + §4.1.3 +
§4.3.1.

### `harness memory list [--scope] [--kind working|semantic|experiential|long_term|multi_agent] [--limit N]`
Reads project memory. Kind taxonomy mirrors paper §3.2.1–§3.2.5.

### `harness memory promote --scope project|session|global --kind <kind> --content "<text>" --run-id <id> --confidence 0..1`
Promotes an evidence-backed entry. Rejected if confidence < 0.4, if
the content matches a secret pattern, or if the kind is unknown.

### `harness memory recall "<query>" [--limit N]`
BM25 keyword search across past run reports. Set
`HARNESS_RECALL_EMBEDDINGS=1` to enable vector recall (when the
embeddings backend is wired).

### `harness sensor list|run <id>`
Lists every applicable sensor for the project and runs them on
demand. Sensors are deterministic (no LLM). The catalog includes:
- Universal: `forbidden_files`, `forbidden_commands`, `secrets_scan`,
  `changed_files`, `performance_budget`
- Per stack (Go/Python/Ruby/Node/Rust/Docker): `go_format`,
  `py_pytest`, `ruby_rubocop`, `node_eslint`, `rust_clippy`, …
- Coverage: `go_coverage_gate` (auto-registered for Go projects)
- Scope: `plan_scope` (auto-registered when an active plan id is
  pinned via `.harness/config/plan.yaml`)

### `harness check`
Runs every applicable sensor but ignores failures (informational).

### `harness ci`
Runs every applicable sensor and exits non-zero on any red. Wired to
the `pre-push` hook by default.

### `harness smoke matrix [--langs csv|all] [--keep] [--json] [--step-timeout DURATION] [--bin path]`
Cross-stack CLI regression: creates a throw-away project per scaffold
and exercises the canonical command sequence (`init`,
`install-git-hooks`, `scaffold apply`, `doctor`, `sensor list`,
`check`, `memory list`, `flow list`, `routes`). Exits non-zero on a
CLI-class failure; tool-class failures (missing host binary) are
reported but tolerated.

### `harness autonomy show|suggest [--apply]`
Inspects and edits the autonomy policy. `suggest` mines past denials
and proposes per-path rules.

### `harness optimize propose|apply --canary <id>|rollback <id>`
Change-contract optimizer per paper §3.5: every proposed mutation must
provide a falsifier test and a rollback command before it is applied.

### `harness evolve diagnose [--json]`
Scans `.harness/logs/events.jsonl`, clusters failures by
(stage, sensor, agent, task, status) signature, and reports cluster
counts. Implements paper §3.5.1 deep telemetry.

### `harness evolve replay <trace-file>`
Scores a candidate trace against the current diagnosis by
text-signature match. A stand-alone baseline.

### `harness evolve sandbox <trace-file> [--candidate <bin>] [--timeout DURATION]`
**Real** A/B replay: runs `harness evolve diagnose` against the trace
inside an isolated workspace once with the baseline binary and once
with the candidate. Reports failure delta and whether the candidate
improved on the baseline. Implements paper §3.5.2.

### `harness evolve propose [--component] [--description] [--rationale] [--risk]`
Records a mutation candidate in `.harness/logs/mutations.jsonl` with
status `proposed`.

### `harness evolve promote <id> --hitl [--reason]`
Human-in-the-loop promotion of a candidate. Refuses without `--hitl`.
Paper §3.5.3 *governed mutation*.

### `harness config show|set|unset|wizard`
Inspects and edits `.harness/config/routes.yaml`. `wizard` is the
interactive flow (prompts you per task type for primary adapter,
fallback chain, budget, and model). Every mutation is appended to
`.harness/logs/config-mutations.jsonl` for audit.

## §4 Scaling — multi-agent orchestration

### `harness agent list|add <id>|login <id>|certify <id>`
Manages adapter registrations. Bundled adapters live under
`internal/app/agentcmd/bundled/*.yaml`; project overrides go in
`.harness/config/agents/*.yaml`.

### `harness route show "<prompt>" [--adapter <id>]`
Explains which adapter the router would pick for a prompt and why.

### `harness routes`
Prints the current task → primary + fallback table.

### `harness skill list|apply <name>`
Bundled skills (e.g. python-test-writer).

### `harness hookpkg install <name>`
Installs bundled lifecycle hooks (gitleaks, conventional-commits, …).
Distinct from `install-git-hooks` (which writes the harness-managed
pre-push, pre-commit, commit-msg scripts).

### `harness mcppkg install <name>`
Installs bundled MCP servers (filesystem, git, sqlite).

## §5 Domain surfaces and supporting commands

### `harness backup save|list|restore <id>`
Tarballs `.harness/` for portable round-tripping.

### `harness containers list`
Inspects detected container runtimes (Docker, Podman, OrbStack).

### `harness runtime status|select`
Inspects and switches the active container runtime.

### `harness images list|audit`
Lists pulled images and audits their digests + provenance.

### `harness secret set|get|list`
Encrypted-at-rest secret store under `.harness/secrets/`.

### `harness audit tail|replay <run-id>`
Tails and replays the append-only event log
(`.harness/logs/events.jsonl`).

### `harness audit-solid [--root <path>]`
Scans for SOLID/god-file violations (file LOC > 400 or imports > 15).

### `harness metrics show|export`
Prints trajectory metrics (ToolCalls, Tokens, Edits, WallMs).

### `harness perf-snapshot`, `harness perf-compare`
Snapshots and diffs performance budgets before/after a change.

### `harness dashboard`
Boots the local React dashboard at <http://localhost:5173>.

### `harness session list|show <id>`
Inspects sessions persisted under `.harness/sessions/<id>.jsonl`.

### `harness completion bash|zsh|fish`
Shell completion scripts.

## Environment variables

| Variable | Effect |
|---|---|
| `HARNESS_PLAIN=1` | Disable ANSI styling |
| `HARNESS_SKIP_CI=1` | Bypass the pre-push gate |
| `HARNESS_SKIP_PRECOMMIT=1` | Bypass the pre-commit hook |
| `HARNESS_SKIP_COMMITMSG=1` | Bypass the commit-msg hook |
| `HARNESS_RELEASE=1` | Allow direct pushes to `main`/`develop` |
| `HARNESS_RECALL_EMBEDDINGS=1` | Enable vector recall backend |

---

## Exit codes

| Code | Meaning |
|---|---|
| 0 | success |
| 1 | command failure (most subcommands) |
| 2 | invalid flag combination |
| Non-zero | propagated from the underlying tool (lint, test, …) |

`harness ci`, `harness coverage`, and `harness smoke matrix` all exit
non-zero on red so they can be wired to git hooks and CI directly.

| `.harness/runs/<id>/report.md` | per-run report |
