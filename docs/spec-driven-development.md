# Spec-driven development (for humans + AI)

HarnessX treats specs as **contracts**. Every change of behaviour ships
with a spec written before the code lands. The spec answers seven
questions in one document: what, why, who, how, risks, evidence, done.

## Where specs live

```
.harness/artifacts/specs/<id-or-slug>.md
```

`<id>` is a ULID when the spec was auto-generated by `harness feature`;
a slug when scaffolded by `harness spec init --name <slug>`. Both share
the same markdown structure — see `internal/spec/spec.go` for the
canonical template.

## Sections (spec §8)

| Section | What goes here |
|---|---|
| Feature Name | One-line title. |
| User Problem | The pain you are eliminating, in user terms. |
| Expected Outcome | The observable result. |
| Scope / Out of Scope | Bounded; what *will* and *will not* change. |
| Business Rules | Domain invariants the change must respect. |
| UX Expectations | Loading / empty / error / disabled states. |
| API Expectations | Endpoints + contract. |
| Data Model Expectations | Tables / columns; flag destructive migrations. |
| Security Considerations | Authn/authz, secrets, audit log, validation. |
| Performance Considerations | Budgets, hotspots, expected p95. |
| Observability Expectations | Logs, metrics, traces, alerting thresholds. |
| Test Plan / E2E Plan | Unit + integration + end-to-end coverage. |
| Acceptance Criteria | Observable conditions that close the spec. |
| Rollback Plan | The undo recipe. |
| Definition of Done | When does this spec retire? |
| Assumptions / Open Questions | Honest uncertainty. |

## Authoring flows

### Greenfield (LLM-assisted)

```bash
harness feature "add product search with filters" --yes
```

Generates `.harness/artifacts/specs/<id>.md` populated with safe
defaults per mode (feature/bugfix/optimization). Edit the markdown
until it reflects the truth, then continue the workflow:

```bash
harness run "implement the product search filters" --execute --yes
```

### Manual (no LLM)

```bash
harness spec init --name product-search-filters --mode feature \
  "add advanced filters to /products"
```

Edit `.harness/artifacts/specs/product-search-filters.md`, commit, link
to the GitHub issue.

## How LLMs read specs

When an agent runs through `internal/app/workflow.executeAgents`, the
context pack `internal/context.Build` produces always carries the
*current spec* in the `CurrentSpec` field (spec §14). Agents are
instructed to:

1. Treat the spec as immutable contract for the run.
2. Ask only when the spec leaves a question explicitly open
   ("Open Questions").
3. Surface any divergence in the final report's "Questions and
   Assumptions" section.

This is what "context engineering" means in HarnessX: instead of
pasting the whole repo, the agent reads a tiny, evidence-backed
context pack pinned to the spec.

## Why specs first

- **Reviewability.** A spec is 200 lines of markdown a reviewer reads
  in 5 minutes. A 2,000-line diff is not.
- **Reproducibility.** Two LLMs given the same spec + context pack
  produce comparable diffs.
- **Memory.** `.harness/artifacts/specs/` becomes the institutional
  record of why each behaviour exists. Bug investigations cross-reference
  it.
- **Safety.** Sensors (`spec_gate`, future) can refuse to execute when
  the spec is missing or contradicts the diff.

## Anti-patterns

- **Implementing first, writing the spec retroactively.** Defeats the
  point. If you skipped the spec, surface that in the PR.
- **Treating Acceptance Criteria as aspirational.** They are
  deterministic — every box should map to a sensor or a test.
- **Hiding open questions.** If the spec ships with assumptions, list
  them. The reviewer needs to know what you decided unilaterally.

## Editing the spec template

The template lives in `internal/spec/spec.go` (constant `tmplBody`).
Edits require:

1. A spec for the spec change (yes, really — see
   `.harness/artifacts/specs/template-evolution.md` once you start).
2. Tests in `internal/spec/spec_test.go` confirming every required
   header still renders.
3. A migration note in `CHANGELOG.md`.
